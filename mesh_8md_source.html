4 # Geometry types
5 There is three kind of geometry representation included in radium source :
6  1. Ra::Core::Geometry::*, which handles geometry data and connectivity as a indexed vertex array.
7 Each vertex is a unique set of position, normal, and other attributes.
8 If indexed, faces are defined with VectorXui, where X is 1 for point, 2 for lines, 3 for triangles, and N for polygons.
9 See inheritance diagram of Ra::Core::Geometry::AbstractGeometry :
10  - Ra::Core::Geometry::AttribArrayGeometry
11  - Ra::Core::Geometry::IndexedGeometry
12  - Ra::Core::Geometry::LineStrip *
13  - Ra::Core::Geometry::PointCloud *
14  - Ra::Core::Geometry::IndexedPointCloud *
15  - Ra::Core::Geometry::LineMesh *
16  - Ra::Core::Geometry::TriangleMesh *
17  - Ra::Core::Geometry::PolyMesh *
18 
19  2. Ra::Core::Geometry::TopologicalMesh, which is an half-edge data structure.
20 A converter allows to go back and forth to `TriangleMesh`
21 without loss of data, but during the conversion, vertices with the same position represents the same topological point (and are hence merged).
22 **Soon deprecated:** The other vertex attributes are stored on half-edges (to manage multiple normals per 3D positions
23 for instance).
24 **New:** The other vertex attributes are stored on wedges. Each half-edge has one wedge index. If multiple half-edge have the same set of attributes (including vertex position) they have the same wedge index at construction. See section [wedges](#wedges) below.
25 
26  3. Ra::Engine::*, which stores a Core Geometry to handle 3D data, and manages the rendering aspect of it (VAO, VBO, draw call).
27 See inheritance digram of Ra::Engine::AttribArrayDisplayable
28  - Ra::Engine::Displayable
29  - Ra::Engine::AttribArrayDisplayable
30  - Ra::Engine::CoreGeometryDisplayable
31  - Ra::Engine::PointCloud *
32  - Ra::Engine::IndexedGeometry
33  - Ra::Engine::IndexedAttribArrayDisplayable
34  - Ra::Engine::LineMesh *
35  - Ra::Engine::Mesh *
36  - Ra::Engine::PolyMesh *
37 
38  `*` : the starred classes are the one you want to instanciate, the other are more for code factoring or abstraction.

40 # Colaboration between Core and Engine
41 
42 A Core Geometry can be used on its own.
43 
44 Engine Geometry must own a Core Geometry, either set at construction, or later with loadGeometry.
45 The Core Geometry ownership is then transfered to the Engine Geometry, and can be accessed by reference with Ra::Engine::Displayable::getAbstractGeometry or Ra::Engine::CoreGeometryDisplayable::getCoreGeometry
46 
47 
48 # Data consistency
49 As soon as a Core Geometry is owned by a Engine Geometry, each data update on the Core Geometry attribute trigger a observator method to mark the corresponding GPU data as dirty.
50 On the next Ra::Engine::CoreGeometryDisplayable::updateGL, the dirty data will be updated on the GPU.

52 # Mesh creation
53 `GeometryComponent` is in charge of loading a `GeometryData` and create the corresponding `Mesh`.
54 
55 The user can add on the fly new vertex attributes to a Ra::Core::Geometry::AttribArrayGeometry
56 An attribute is defined by a unique name (std::string) and then represented as a Ra::Core::Utils::AttribHandle.
57 The type of the attribute is defined by the caller using the template parameter of the method Ra::Core::Geometry::AttribArrayGeometry::addAttrib.
58 To ensure consistency, it is strongly recommended to store the handle returned by the Ra::Core::Geometry::AttribArrayGeometry::addAttrib method:
59 ~~~{.cpp}
60  auto handle1 = m.addAttrib<Vector3>( "vector3_attrib" );
61 ~~~
62 Attribute names `in_position` and `in_normals` are reserved.
63 Note that handles to existing attributes can be retrieved by name (see Ra::Core::Geometry::AttribArrayGeometry::getAttrib), however
64 the caller have to provide the type of the associate attribute.
65 There is no type check on this access, and we recommend not to use this. We keep this method for convenience in some
66 specific cases such as copy of attributes, however note that it might be marked as deprecated at any time.
67 
68 There is no (in the current version) data checking, and attributes are simple `std::vector`s added to the attribute
69 manager. It's meant to be used as vector of data, such as vertices and normals, with the same size and accessed with
70 the vertices indices. But please be aware no check nor allocation are done, so the allocation has to be performed on
71 client side.
72 
73 When copying a Ra::Core::Geometry::AttribArrayGeometry or one of its descendent class, all its attributes are copied, unless it is done through Ra::Core::Geometry::AttribArrayGeometry::copyBaseGeometry, for which no attribute is copied.
74 In order to copy some/all of the attributes, the dedicated methods must be used.
75 
76 Attribute writes must be with "lock" to ensure data syncronisation.
77 Utility methods are provided for position and normals :
78 
79  - Ra::Core::Geometry::IndexedGeometry::setIndices
80  - Ra::Core::Geometry::IndexedGeometry::getIndicesWithLock
81  - Ra::Core::Geometry::IndexedGeometry::indicesUnlock
82  - Ra::Core::Geometry::AttribArrayGeometry::setVertices
83  - Ra::Core::Geometry::AttribArrayGeometry::verticesWithLock
84  - Ra::Core::Geometry::AttribArrayGeometry::verticesUnlock
85  - Ra::Core::Geometry::AttribArrayGeometry::setNormals
86  - Ra::Core::Geometry::AttribArrayGeometry::normalsWithLock
87  - Ra::Core::Geometry::AttribArrayGeometry::normalsUnlock
88 
89 Others attributes are written with :
90  - Ra::Core::Geometry::AttribArrayGeometry::getAttrib to get an attrib handle
91  - Ra::Core::Utils::Attrib::setData to set the data directly
92 or
93  - Ra::Core::Utils::Attrib::getDataWithLock to get a writable reference to the data array
94  - Ra::Core::Utils::Attrib::unlock to release the writable reference
95 
96 For instance
97 
98 ~~~{.cpp}
99 using Ra::Core::Geometry::TriangleMesh;
100 TriangleMesh m;
101 TriangleMesh::PointAttribHandle::Container vertices;
102 TriangleMesh::NormalAttribHandle::Container normals;
103 
104 vertices.push_back( {0, 0, 0} );
105 vertices.push_back( {1, 0, 0} );
106 vertices.push_back( {0, 2, 0} );
107 normals.push_back( {0, 0, 1} );
108 normals.push_back( {0, 0, 1} );
109 normals.push_back( {0, 0, 1} );
110 
111 m.setVertices( std::move( vertices ) );
112 m.setNormals( std::move( normals ) );
113 
114 m.setIndices( {0, 1, 2} );
115 
116 auto handle1 = m.addAttrib<Vector3>( "vector3_attrib" );
117 auto& attrib1 = m.getAttrib( handle1 );
118 auto& buf = attrib1.getDataWithLock();
119 
120 buf.reserve( 3 );
121 buf.push_back( {1, 1, 1} );
122 buf.push_back( {2, 2, 2} );
123 buf.push_back( {3, 3, 3} );
124 attrib1.unlock();
125 
126 auto handle2 = m.addAttrib<float>( "float_attrib" );
127 auto& attrib2 = m.getAttrib( handle2 );
128 attrib2.setData( {1.f, 2.f, 3.f} );
129 
130 TriangleMesh m2;
131 m2.copyBaseGeometry( m );
132 m2.copyAttributes( m, handle1 );
133 ~~~

138 ## Wedges {#wedges}
139 
140 ![Wedge concept in a nutshell.](wedges.svg)
141 
142 Wedge are built at construction by using CoreMesh vertex index (which are supposed to represent wedges). Vertices with same (exact) position are merged topologically.
143 The figure above show the basic concept of wedges for the example of vertex with color attribute (same works for normals, texture coordinates or any other attributes).
144 a) A colored mesh, each vertex can have a different color depending on which face is considered. b) for rendering, vertex are duplicated the needed number of time. Here the center vertex is duplicated three time, while the top right one is duplicated two times. c) But for topological computation, one need to know these vertex are actually the same position, and are modified the same way. d) So the attributes (here color only) are represented as wedges associated with each coherent set of position+attributes. Here the center vertex has three wedges, while the top right vertex has two wedges. e) In the TopologicalMesh implementation, wedges are stored in an array, independently of the topology. Each half-edge has a wedge index. Half-edges that share the same wedge have the same index.
145 During manipulation, if two wedges become the same (i.e. for the example on the figure, if we recolor the whole mesh using a single color), **they are not merged**.
146 To merge wedges an explicit call to TopologicalMesh::mergeEqualWedges is needed.
147 When a wedge is not referenced anymore (because referencing halfedges have been deleted), it is marked for deletation. When one call Ra::Core::Geometry::TopologicalMesh::garbage_collection(), the marked wedges are eventually deleted and halfedges index are updated accordingly.
