4 Custom plugins can be easily added to Radium ecosystem.
5 In addition to the Ra::Plugins::RadiumPluginInterface API,
6 Radium provides a set of tools simplifying the deployment of plugins, which among
7 other things take care of the portability and relocatability of the
8 binaries.
9 
10 This manual describes how to configure, compile and use your plugin with any Radium-Plugin compatible application.
11 
12 # Setting a CMakeLists.txt for a Radium plugin
13 
14 To compile and use a plugin,
15  the Radium environment must be configured,
16 compiled and installed beforehand.
17 
18 The main CMakeLists.txt file for compiling a Radium plugin
19 could be as simple as
20 
21 ```cmake
22  cmake_minimum_required(VERSION 3.6)
23  #------------------------------------------------------------------------------
24  # Policies and global parameters for CMake
25  if (POLICY CMP0077)
26  cmake_policy(SET CMP0077 NEW)
27  endif ()
28  set(CMAKE_DISABLE_SOURCE_CHANGES ON)
29  set(CMAKE_DISABLE_IN_SOURCE_BUILD ON)
30  #------------------------------------------------------------------------------
31  # Project definition
32  project(MyPlugin VERSION 1.0.0 LANGUAGES CXX)
33 
34  # Use installed Radium environment
35  find_package(Radium REQUIRED Core Engine PluginBase)
36 
37  # Find and configure Qt environment (Radium requires Qt >= 5.1)
38  find_package(Qt5 COMPONENTS Core REQUIRED)
39  set(Qt5_LIBRARIES Qt5::Core)
40  set(CMAKE_AUTOMOC ON)
41  set(CMAKE_AUTORCC ON)
42 
43  # list all the sources and headers of the Plugin
44  set(sources /*... */)
45  set(headers /*... */)
46 
47  set(CMAKE_INCLUDE_CURRENT_DIR ON)
48 
49  # A Plugin is a library that uses these sources and headers.
50  add_library( ${PROJECT_NAME} SHARED ${sources} ${headers} )
51 
52  target_include_directories(${PROJECT_NAME} PRIVATE ${CMAKE_CURRENT_SOURCE_DIR}/src)
53  target_compile_definitions(${PROJECT_NAME} PRIVATE "-D${PROJECT_NAME}_EXPORTS")
54 
55  target_link_libraries(${PROJECT_NAME} PUBLIC
56  Radium::Core Radium::Engine Radium::PluginBase
57  ${Qt5_LIBRARIES} )
58 
59  # Configure the plugin deployement using the Radium configuration tool
60  configure_radium_plugin( NAME ${PROJECT_NAME} )
61 ```
62 
63 Such a CMakeLists.txt is as ususal as others. The only command that is specific to Radium plugins is the last line
64 `configure_radium_plugin( NAME ${PROJECT_NAME} )` that aims at configuring the plugin installation along with
65 its associated resources.
66 
67 The command `configure_radium_plugin`takes several parameters.
68 The full usage of this command is :
69 
70 ```cmake
71 configure_radium_plugin(
72  NAME nameOfTheTargetOtInstall # mandatory
73  INSTALL_IN_RADIUM_BUNDLE # optional
74  RESOURCES ListOfResourcesDirectory # optional
75  HELPER_LIBS ListofHelperLibraries # optional
76  )
77 ```
78 
79 - `NAME nameOfTheTargetOtInstall`: this parameter, mandatory, will configure how to install the target
80  `nameOfTheTargetOtInstall`. This target must correspond to a configured target such the one obtained by
81  `target_link_libraries(nameOfTheTargetOtInstall PUBLIC Radium::Core Radium::Engine Radium::PluginBase ${Qt5_LIBRARIES} )`
82 
83 - `RESOURCES ListOfResourcesDirectory` : this parameter, optional, will install several resources, needed by the plugin for its correct execution,
84  so that the Radium resource locator system will be able to find them.
85  Resources could be shader source files, images, data files, etc...
86 
87 - `HELPER_LIBS ListofHelperLibraries`: this parameter, optional, will allow to fetch and install, alongside the plugin,
88  libraries and their resources used by the plugin and not available in the Radium Bundle. This is useful when a
89  plugin just defines the radium application interface to services offered in an external, radium dependent, library.
90 
91 - `INSTALL_IN_RADIUM_BUNDLE` : this parameter, optional, will install the plugin and its associated resources directly
92  into the Radium Bundle installation directory. This will allow to have Plugins relocatable at the same time than
93  the Radium Bundle. If this parameter is given, it replace the install location configuration that uses
94  the cmake standard option `-DCMAKE_INSTALL_PREFIX=whereToInstallThings`
95 
96 Note that, as it is the case for applications, Plugins does not need to be installed to be used by any applications.
97 Plugins could be loaded and used from their build-trees.

99 # Configuring the plugin
100 
101 Configuring the plugin for compilation and install depends on several use cases that might evolve
102 during the plugin development.
103 
104 There are three main uses cases we can identify :
105 
106 1. You are developing/testing your plugin. There is no need to install the plugin and no install option is needed when configuring using cmake.
107  Any plugin compatible application might find and use it from its build tree (e.g `build-release/Plugins`).
108  For this, you have to configure once your application by adding the build path of the plugin
109  to the plugin search path of the application. Each time you will launch this application, it will search in all its
110  registered search paths for compatible plugins.
111  The advantage of not installing plugins while developping/testing is a fast startup of the application
112  and efficient update of the Plugin resources as Resources directories are _linked_ (symbolic links on systems that
113  support them) into the build tree.
114 
115 2. You have finished to develop the plugin, and will only occasionally recompile it for maintenance. If you developed several plugins, for your own usage, and you want your
116  plugin compatible applications to find and load them quickly,
117  you might install your plugins in one unique location and tell the applications to load plugins from
118  there. To do so, use the cmake configuration option `-DCMAKE_INSTALL_PREFIX=thePathToTheInstalledPlugins` and do a `make install`
119  You can then configure your applications so that they will search plugins from the `thePathToTheInstalledPlugins` directory.
120  Each time the application will start, all the plugins from this `thePathToTheInstalledPlugins` will be
121  loaded, even those installed here after the configuration of the plugin location in the application.
122 
123 3. You want to distribute plugins binaries alongside the Radium bundle.
124  In this case, you have to configure the plugin with the`INSTALL_IN_RADIUM_BUNDLE`option and do a `make install` to
125  install the plugins in the Radium Bundle. Note that this requires write access to the Radium Bundle.
126 
127 
128 Once you have chosen the adequate use case, you can configure the plugin using cmake.
129 
130 As we disable "in source" build in the CMakeLists.txt example above, we need to create a build directory:
131 
132 ```Shell
133 mkdir build-release && cd build-release
134 ```
135 
136 With Radium-Engine installed in the directory `pathToRadiumInstallation`,
137 configure and compile the project with :
138 
139 ```Shell
140 cmake ../ -DRadium_DIR=pathToRadiumInstallation/lib/cmake/Radium
141 cmake --build .
142 ```
143 
144 If you just want to use the plugin directly from its build-tree, the resulting dynamic library will be found in the
145 `build-release/Plugins/lib` directory and the associated resources will be linked (or copied on systems that do not
146 support symbolic links) in the directory `build-release/Plugins/Resources`. In this case, note that the plugin from the
147 buildtree could generally not be moved as it is linked to used libraries using relative paths.
148 
149 If you install the plugin, execute the command `cmake --install`
150 that will install the plugin according to your configuration choice as described above (default installation destination, user defined installation directory or Radium Bundle installation).
151 The dynamic library will then be copied into `pathToInstallPlugins/lib` and the Resources will be copied into
152 `pathToInstallPlugins/Resources`.
153 
154 
155 Remember that resources associated with the plugin using the option `RESOURCES ListOfResourcesDirectory` of the
156 `configure_radium_plugin` command are _linked_ into the buildtree (on systems supporting symbolic links)
157 and _copied_ into the installed locations.

159 # Using the Plugin
160 
161 Any application that inherits from `Ra::GuiBase::BaseApplication` could use any Plugin developped using the
162 Radium Engine.
163 As plugins are Qt objects that implement specific interface, it is also permitted to have any Qt
164 Application to use plugins. Meanwhile, the following documentation is related to the Radium BaseApplication
165 derived applications.
166 
167 If the plugin is installed in the Radium Bundle directory, it will be automatically loaded by any plugin-aware
168 application.
169 But, when developping a plugin, the installation in the Radium bundle directory is not always permitted nor efficient.
170 
171 It could then be useful to use the Plugin from its own installation directory or directly from its build-tree.
172 To do that, the application could register the plugin directory location by calling the
173 `Ra::GuiBase::BaseApplication::addPluginDirectory( const std::string& pluginDir );` method that will record, in the
174 QSettings file associated with the application, the directory in which the Radium Engine will look for plugins.
175 Once this directory is registered, any plugin that will be found in this directory will be loaded at each application
176 startup if the build type of the plugin is compatible with the build type of the application (release, debug, ...).
177 If the plugin location that is registered correspond to the build tree location of the plugin, there is no need to
178 instal it.
179 
180 Note that the main Radium demonstration application (`main-app`) offer a menu entry
181 (`File/Plugins/Add plugin path)` that call the registration method once the user has selected a plugin location.
