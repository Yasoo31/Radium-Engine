4 # Rendering in Radium
5 Rendering in Radium is managed by the abstract Ra::Engine::Renderer class that provides the following main rendering method.
6 
7 ## Main abstract render method
8 
9 This method is general and cannot be modified, it always does the following
10 
11 ~~~{.cpp}
12 void Ra::Engine::Renderer::render( const Ra::Engine::ViewingParameters& data )
13 {
14  // 0. Save an eventual already bound FBO
15  saveExternalFBOInternal();
16 
17  // 1. Gather render objects and update them
18  renderObjects = m_engine->getRenderObjectManager()->getRenderObjects();
19  updateRenderObjectsInternal( data, renderObjects );
20 
21  // 2. Feed render queues
22  feedRenderQueuesInternal( data, renderObjects );
23 
24  // 3. Do picking if needed
25  if ( !m_pickingQueries.empty() )
26  {
27  doPicking();
28  }
29 
30  // 4. Do the rendering.
31  renderInternal( data );
32 
33  // 5. Post processing
34  postProcessInternal( data );
35 
36  // 6. Write final texture to framebuffer / backbuffer
37  // (depending on wether anything was saved on step 0. or not)
38  drawScreenInternal();
39 }
40 ~~~
41 The meaning of each rendering steps is described below.
42 
43 ### 0. Save an eventual already bound FBO
44 In some cases (like with `QtOpenGLWidget`), you do not draw directly on the screen, but you have instead
45 to feed an already bound FBO. Since the default renderer uses multiple FBOs, the Qt's one must be saved.
46 
47 _This behaviour cannot be modified._
48 
49 ### 1. Gather render objects and update them
50 This construct the set of objects that must be drawn for the current frame and update their OpenGL state.
51 
52 ### 2. Feed the render queue
53 This construct the set of render actions that must be done for the current frame.
54 
55 ### 3. Do picking if needed
56 If there has been some picking requests since the last frame, `doPicking` is called.
57 This function just renders all the objects (except _debug_ ones) by drawing them in some color given the ID
58 of the entity a render object is attached to.
59 
60 Then, for each picking request done, `glReadPixels` is called at the requested location, and object ID is retrieved.
61 
62 ### 4. Do the rendering
63 This method does most of the whole rendering work
64 and outputs one final _render pass_ texture, ready to be post-processed.
65 
66 ### 5. Post-process the rendering result
67 This method will apply post-processing filters on the computed image.

70 ## Defining a concrete renderer
71 
72 To define a concrete renderer, the Ra::Engine::Renderer class can be inherited to define, at least, the
73 `renderInternal` method. Note that only two methods can be overridden for the renderer, renderInternal (step 4) and
74 postProcessInternal (step 5).
75 
76 A concrete renderer will build its rendering loop, the `renderInternal` method, by decomposing it into passes
77 configured, for each Ra::engine::RenderObject, using the Ra::Engine::RenderTechique class.
78 
79 A Ra::Engine::RenderTechnique consists in a set of named Ra::Engine::ShaderConfiguration that provide the GLSL code used
80 by each pass to compute the appearance of a Ra::Engine::RenderObject and a set of Ra::Engine::ShaderParameterProvider
81 that provide shader data to be used for each pass.
82 
83 When rendering, the renderer will loop over all the configured Ra::Engine::RenderObject and, for each rendering pass,
84 will get the OpenGL pipeline configuration and parameters from the Ra::Engine::RenderTechnique associated with the
85 Ra::Engine::RenderObject.
86 
87 The Radium Engine exports a ready to use renderer, Ra::Engine::ForwardRenderer, that decomposes the rendering into three
88 passes as described in the \ref forwardrendererconcept document.
