4 A Ra::Engine::Material is a way to control the appearance of an object when rendering.
5 A Ra::Engine::Material defines the _Bidirectional Scattering Distribution function (BSDF)_ to be applied on an object.
6 
7 A material is associated to the render geometry of an object (a Ra::Engine::RenderObject) through a
8 Ra::Engine::RenderTechnique.
9 
10 A Ra::Engine::RenderTechnique describes how to use a Ra::Engine::Material to render an object in OpenGL and consists in
11 a set of named Ra::Engine::ShaderConfiguration that encompass the OpenGL representation of a Material (glsl code and
12 associated data) with, at least, vertex and fragment shaders. The name of each Ra::Engine::ShaderConfiguration
13 corresponds to the way a Ra::Engine::Renderer manages its rendering loop.
14 
15 A Ra::Engine::ShaderConfiguration is associated with a Ra::Engine::ShaderProgram bound by the Ra::Engine::Renderer
16 when rendering an object.
17 
18 The Radium Engine exposes some predefined materials, the _Radium Material Library_, with render techniques corresponding
19 to the Ra::Engine::ForwardRenderer default renderer.
20 
21 # Content of the Radium Material Library
22 The Radium Material Library defines two default material :
23  - BlinnPhong, Ra::Engine::BlinnPhongMaterial, corresponding to the Blinn-Phong BSDF.
24  - Plain, Ra::Engine::PlainMaterial, corresponding to a diffuse, lambertian BSDF.
25 
26 The _Radium Material Library_ can be used as this by any Radium Application or can be extended by an application or a
27 Radium Plugin by implementing the corresponding interfaces as described in the
28 [Extending the _Radium Material Library](#extend-mtl-lib).
29 
30 For each material of the library, a default Ra::Engine::RenderTechnique, corresponding to the standard usage of the
31 material by the Ra::engine::ForwardRenderer and an optional Ra::Engine::EngineMaterialConverters::ConverterFunction,
32 used when loading files to convert file representation of a Material to Radium representation of this material, are
33 made available through the dedicated factories Ra::Engine::EngineRenderTechniques and
34 Ra::Engine::EngineMaterialConverters.
35 
36 See the [Material registration into the Engine](#registration-mtl-lib) section of this documentation to learn more about
37 these factories.

39 # Simple usage of materials {#simple-mtl-usage}
40 
41 A simple usage of material is demonstrated in the [HelloRadium Application](https://github.com/STORM-IRIT/Radium-Apps).
42 
43 When building scene to render, a Ra::Engine::Component must be added to a system as described into the
44 _Radium Engine programmer manual_.
45 A component holds one or several Ra::Engine::RenderObject that will be drawn when rendering.
46 
47 To define a Ra::Engine::RenderObject and add it to the component, the geometry of a 3D object (a Ra::Engine::Mesh)
48 must be associated with a Ra::Engine::RenderTechnique that links to the required Ra::Engine::Material.
49 
50 To do that, the following steps must be done :
51 1. Create the Ra::Engine::Mesh (see the [documentation about Meshes](@ref develmeshes))
52 \snippet HelloRadium/minimalradium.cpp Creating the Engine Mesh
53 
54 2. Create the Ra::Engine::Material
55 \snippet HelloRadium/minimalradium.cpp Creating the Material
56 
57 3. Create the Ra::Engine::RenderTechnique, here using the Ra::Engine::EngineRenderTechniques factory, and associate it
58 to the material
59 \snippet HelloRadium/minimalradium.cpp Creating the RenderTechnique
60 
61 4. Create the Ra::Engine::RenderObject and add it to the Ra::Engine::Component
62 \snippet HelloRadium/minimalradium.cpp Creating the RenderObject
63 
64 Note that this way of using the _Radium Material Library_ is very related to the default Radium rendering capabilities
65 exposed by the [Radium forward renderer](@ref forwardrendererconcept).
66 See the [Render technique management](./rendertechnique) documentation to learn how to create your own
67 Ra::Engine::RenderTechnique, potentially without associated material.
68 
69 If one wants to render objects without BSDF computation but with a specific color computation for the fragment,
70 follow the guidelines from [the dedicated section](#non-bsdf-rendering) of this documentation.

72 # Extending the Radium Material Library {#extend-mtl-lib}
73 
74 The _Radium Material Library_ could be extended to handle several _Bidirectional Scattering Distribution function_.
75 
76 In order to make these extensions available to each Radium developer, the _Radium Material Library_ defines several
77 interfaces and factories for material management.
78 
79 A material is defined by two programming interfaces. The Ra::Engine::Material that defines the
80 [C++ interface](#cpp-mtl-lib) made available for applications and plugins and a [GLSL interface](#glsl-mtl-lib) that
81 allows shader reuse and composition for OpenGL rendering.
82 
83 The C++ interface is implemented in a `NameOfTheMaterial.hpp/.cpp` source file.
84 
85 The GLSL interface is composed of several parts :
86 1. The implementation of a [BSDF interface](#bsdf-interface) and a [micro-geometry interface](#microgeometry-interface)
87 in a `NameOfBSDF.glsl` file that will be included in every fragment shaders that need the implementation of the bsdf.
88 2. The implementation of one or several vertex shaders that will compute the data used by the
89 [vertex attrib interface](#vrtx-attr-interface). These shaders are renderer-specific and the programmer must at least
90 give one for the Radium default renderer.
91 3. The implementation of one or several fragment shaders that, by using the
92 [vertex attrib interface](#vrtx-attr-interface), will compute the final color of a fragment using the
93 [BSDF interface](#bsdf-interface) and the [micro-geometry interface](#microgeometry-interface). These shaders are
94 renderer-specific and the programmer must at least give one for the Radium default renderer.
95 
96 ## C++ interface {#cpp-mtl-lib}
97 The Ra::Engine::Material interface defines the internal abstract representation of a Material.
98 This interface defines all the methods required to parametrized the OpenGL pipeline for rendering and will be used
99 mainly by the Ra::Engine::RenderTechnique and the Ra::Engine::Renderer classes.
100 
101 When implementing this interface, and in order to make the material available to all applications or plugins,
102 two static methods to register and unregister the material into the _Radium Material Library_ must also be developed.
103 These method will populate the _Radium Material Library_ factories with specific helper functions to use the material
104 in the default Radium forward renderer.
105 Mainly, the `registerMaterial()` method will record an helper function to build a material-related
106 Ra::Engine::RenderTechnique dedicated to the Radium forward renderer.
107 ~~~{.cpp}
108 class MyMaterial : public Ra::Engine::Material {
109 public:
110  // implementation of the abstract interface
111  ...
112 private:
113  // data member for MyMaterial
114  ...
115 public:
116  static void registerMaterial();
117  static void unregisterMaterial();
118  // MyMaterial specific public interface
119  ...
120 }
121 ~~~
122 See the [Render technique management](./rendertechnique) for documentation on how to build such an helper function.

124 ## GLSL interface {#glsl-mtl-lib}
125 Being able to compose shaders in a specific renderer while taking profit of Radium Material Library
126 (either included in the base engine or defined in plugins) require a clean definition of appearance computation
127 process and the definition of a glsl interface.
128 
129 
130 ## Appearance computation needs
131 In order to compute the appearance of an object, and according to the OpenGL/GLSL approach of rendering, several
132 aspects might be taken into account and might be integrated into the interface definition to make a material
133 renderer-agnostic.
134 
135 The user or the implementer could rely on default implementation, provided by the _Radium Material Library_ of most of
136 these functionalities and concentrate its effort in developing the required functionalities for its material/renderer.
137 
138 1. Accessing or computing [appearance attributes that depends on geometric data](#vrtx-attr-interface), such as
139  - vertex attribs (normal, position, color, ...)
140 2. Accessing or computing [microgeometric attributes](#microgeometry-interface) that depends on extension of geometric data, such as
141  - Normal maps
142  - Displacement maps
143  - Transparency maps
144 3. Accessing and [computing appearance attributes that depends on the BSDF model](#bsdf-interface) used to render the object, such as
145  - Blinn-Phong BSDF (default BSDF in Radium Engine)
146  - Microfacet - based BSDF (added by plugins such as GLTF-2 or PBRT)
147 
148 ### Vertex attrib interface {#vrtx-attr-interface}
149 In order to compute the appearance of an object, one need to rely on parameters defined directly on the geometry of
150 the object. Such parameters (position, normal, tangent, ...) are passed to the shader systems as vertex attributes.
151 
152 In order to keep the appearance computation agnostic on the way vertex attribs are named or accessed, we must
153 propose an abstract interface. But, and this is particular to these attributes, one can access to the attributes
154 himself, on the vertex, or to the attributes interpolated by the rasterizer, on the fragment.
155 Accessing the Attribute directly on the vertex (i.e. on a vertex shader) does not necessitate an interface as
156 each shader must define its attributes and as the [Mesh API](./mesh.md) allows to communicate between C++ and GLSL.
157 
158 Note that the attributes accessed through the Vertex attrib interface **must** be defined in world space.
159 Even if not necessarily efficient (some transformations might be computed twice), this will ensure more simple
160 lighting computation.
161 This might be changed in future version of Radium.
162 
163 Each glsl component that needs to access to fragment-interpolated attributes must do that through this interface.
164 The default declaration and implementation of this fragment interface is given in the file
165 `Shaders/Materials/VertexAttribInterface.frag.glsl`. This interface might be included in any **fragment shader** that
166 want to use it by `#include "VertexAttribInterface.frag.glsl"`.
167 
168 It relies on some standard vertex attribute that have to be set by the vertex shader with respect to the following
169 out binding :
170 ~~~{.cpp}~
171 // All attributes are given in world space
172 layout (location = 0) out vec3 position;
173 layout (location = 1) out vec3 normal;
174 layout (location = 2) out vec3 texcoord;
175 layout (location = 3) out vec3 vertexcolor;
176 layout (location = 4) out vec3 tangent;
177 ~~~
178 The default implementation of the fragment interface is robust to inactive attributes, i.e. attributes that are not set
179 by the vertex shader.
180 
181 If one wants to implement its own vertex attribute interface, in order, e.g., to take advantage of some application
182 specific data layout, the following declaration of the glsl functions exported by the interface must be done.
183 The implementation of the interface could then be developed in the appropriate glsl file.
184 Note that not all the functions must be pre-declared and the programmer could restrict himself to the only functions
185 he needs :
186 
187 ~~~{.cpp}
188 // Return the Vertex position, in world space, defined or interpolated from vertices
189 vec4 getWorldSpacePosition();
190 
191 // Return the geometric normal, in world space, defined or interpolated from vertices
192 vec3 getWorldSpaceNormal();
193 
194 //Return the geometric tangent, in world space, defined or interpolated from vertices
195 vec3 getWorldSpaceTangent();
196 
197 //Return the geometric bi-tangent, in world space, defined or interpolated from vertices
198 vec3 getWorldSpaceBiTangent();
199 
200 //Return the 2D parametric position of the vertex (the texture coords), defined or interpolated from vertices
201 vec3 getPerVertexTexCoord();
202 
203 // Return the base color, defined or interpolated from vertices
204 vec4 getPerVertexBaseColor();
205 
206 // return the specular color, defined or interpolated from vertices
207 vec3 getPerVertexSpecularColor();
208 ~~~
209 
210 Note also that if a function is not needed by a shader, there is no need to implement its interface.

213 ### Microgeometry interface {#microgeometry-interface}
214 Defining the micro-geometry procedurally or by using textures allows to de-correlates the geometric sampling from the
215 appearance parameters sampling.
216 The best example of procedural micro-geometry is normal mapping.
217 
218 For a practical introduction to this kind of approach with opengl, the reader could refer to the
219 [Learn opengl tutorial](https://learnopengl.com/).
220 
221 For a more indepth presentation of these kind of techniques for realtime rendering, we encourage the reader to refer to
222 [Real-Time Rendering, Fourth Edition, by Tomas Akenine-Möller, Eric Haines, Naty Hoffman, Angelo Pesce, Michał Iwanicki
223  and Sébastien Hillaire](https://www.realtimerendering.com/).
224 
225 The microgeometry could also define which fragment is transparent.
226 So, in order to be able to compute or discard transparent fragments, one need to define a `toDiscard` function.
227 
228 The interface (to be implemented in the file `name_of_the_BSDF.glsl`) is then
229 
230 ~~~{.cpp}
231 // Return the world-space normal computed according to the microgeometry definition`
232 // If no normal map is defined, return N
233 vec3 getNormal(Material material, vec3 texCoord, vec3 N, vec3 T, vec3 B);
234 // return true if the fragment must be condidered as transparent (either fully or partially)
235 
236 bool toDiscard(Material material, vec4 color);
237 ~~~

240 ### BSDF interface {#bsdf-interface}
241 Implementing or using the GLSL BSDF interface is based on the fact that the method Ra::Engine::Material::getMaterialName()
242  must return a string that contains the `name_of_the_BSDF` implemented in a file named `name_of_the_BSDF.glsl`.
243 This file is preloaded at [material registration](#registration-mtl-lib) into a `glNamedString` to allow inclusion by others.
244 
245 In order to be composable by Radium applications and renderers, this glsl file must only contains the implementation
246 of the BSDF interface, with no `void main(){...}`, the implementation of the micro-geometry interface and must only
247 access to vertex attribs by using the vertex attribute interface or so on ...
248 
249 This file must contain an inclusion guard :
250 ~~~{.cpp}
251 #ifndef METALLICROUGHNESS_GLSL
252 #define METALLICROUGHNESS_GLSL
253 .
254 .
255 .
256 #endif
257 ~~~
258 
259 The BSDF interface consists in the following
260 
261 ~~~{.cpp}
262 // Concrete definition of the Material structure that contains the BSDF parameters
263 struct Material {
264  ...
265 };
266 
267 // Returns the base color, or albedo, of the material, at the surface coordinates defined by texCoord.
268 // The returned color will eventually be used to infer if the fragment is transparent or not.
269 // The alpha channel could then vary from 0 (totally transparent) to 1 (totally opaque)
270 vec4 getBaseColor(Material material, vec3 texCoord);
271 
272 // Returns the so called "Diffuse Color" of the material, at the surface coordinates defined by texCoord.
273 // This could be the same that the base color or obtained by a more or less complex computation (Fresnel, ...)
274 // The alpha channel is the same than the one computed for getBaseColor().
275 // Note : might be deprecated in a near future (see getSeparateBSDFComponent)
276 vec4 getDiffuseColor(Material material, vec3 texCoord);
277 
278 // Returns the so called "Specular Color" of the material, at the surface coordinates defined by texCoord.
279 // This could be the same that the base color (without the alpha channel) or obtained by
280 // a more or less complex computation
281 // Note : might be deprecated in a near future (see getSeparateBSDFComponent)
282 vec3 getSpecularColor(Material material, vec3 texCoord);
283 
284 // Return the reflectance value for the material, at surface coordinates defined by texCoord,
285 // for the incoming and outgoing directions `wi` and `wo`.
286 // the reflectance value is defined as bsdf * cos(Theta_i).
287 // IMPORTANT : for efficiency reasons, these directions MUST be in local frame.
288 // The local Frame is the Frame wher the Geometric normal is the Z axis,
289 // and the tangent defined the X axis.
290 vec3 evaluateBSDF(Material material, vec3 texCoord, vec3 wi, vec3 wo);
291 
292 // Extract separated components of the bsdf value for the material according to the Light/View/Normal configuration.
293 // This function must return 0 if the bsdf is not defined for the given configuration (e.g. back-face lighting) and 1
294 // if the results are usable.
295 // Separated components are the diffuse part of the BSDF and the specular part of the BSDF.
296 // NOTE : these separated components must not include the cos(Theta_i).
297 // NOTE : For non separable bsdf model (e.g. bsdf encoded on shperical Harmonics), it is recomended to set the diffuse
298 // part to 0 and to evaluate the bsdf in the specular part.
299 int getSeparateBSDFComponent(Material material, vec3 texCoord, vec3 Light, vec3 View, vec3 Normal, out vec3 diffuse, out vec3 specular);
300 
301 // Return the GGX equivalent "roughness" of the BSDF as defined in the GGX microfacet bsdf model.
302 // Roughness might range from 0.04 to 1 for specular to diffuse materials.
303 // This roughness might be used, e.g. to filter environment map when computing environment lighting.
304 float getGGXRoughness(Material material, vec3 texCoord);
305 ~~~

307 ### Emissivity interface {#emissivity-interface}
308 Some materials are not only reflective, hence implementing the BSDF interface, but also can be emissive.
309 To allow a renderer to access the emissivity of a material the following GLSL function must
310 defined in the same GLSL file than the BSDF and microgeometry interface :
311 ~~~
312 // Return the emissivity of the material
313 vec3 getEmissiveColor(GLTFCommon material, vec3 textCoord);
314 ~~~

316 ## Material registration into the Engine {#registration-mtl-lib}
317 When implementing the [GLSL interface](#glsl-mtl-lib) of a Material, the user can rely on several
318 glsl components defined by the Engine. Glsl components are helper functions and data structure that could
319 be used to develop specific shaders.
320 
321 To make this common component available to users, and also each user defined component that want to be available to
322 others, the Radium Engine defines a material component registration system that allows to populate and extend the
323 _Radium Material Library_.
324 
325 The registration and glsl component access system is made of 3 parts
326 1. Registration of OpenGL/GLSL named string : defining a re-usable GLSL component.
327 2. Registration of OpenGL/GLSL Program configuration : defining how to link shaders into a program for a specific
328 rendering step.
329 3. Render Technique factory : defining which Program configurations must be used for each renderer-specific task.
330 
331 a fourth part, optional, could be defined to convert a Ra::Core::Asset:MaterialData (file representation of a material)
332 to a Ra::Engine::Material.
333 
334 ### Registration of OpenGL/GLSL named string
335 Relying on already developed GLSL component require that this component could be included in the client code.
336 In GLSL, this is done using the preprocessor directive ``#include </ComponentPath/ComponentName.glsl>``.
337 As specified by the ARB_shading_language_include specification, included files must be preloaded in a
338 GlNamedString object. The Ra::Engine::ShaderManager provide a way to populate the GLNamedString ecosystem.
339 
340 To register a GLSL component, from a file named ``SharedComponent.glsl``, so that it could be included in others GLSL files
341 by ``#include </SharedComponent.glsl>``, one just need to do the following
342 
343 ~~~{.cpp}
344  // adding the material glsl implementation file
345  ShaderProgramManager::getInstance()->addNamedString(
346  "/SharedComponent.glsl", "/absolute/path/to/file/SharedComponent.glsl" );
347 ~~~
348 
349 ### Registration of OpenGL/GLSL Program configuration
350 The use of GLSL component to render object requires the building of a render-task specific OpenGL Program that link
351 together the GLSL component and shaders addressing stages of the OpenGL Pipeline. According to the OpenGL specification,
352 A vertex shader and a fragment shader stage are mandatory whereas tesselation end geometry shader are optional and
353 depends only of the way one want to configure its pipeline for rendering.
354 
355 To describe the OpenGL program configuration, and make it reusable, the configuration must be registered in the
356 Ra::Engine::ShaderConfigurationFactory. To do that, for each reusable configuration, one just need to do the following
357 ~~~{.cpp}
358  // build the configuration
359  Ra::Engine::ShaderConfiguration myConfig(
360  "ConfigName",
361  "absolute/path/to/vertexshader.vert.glsl",
362  "absolute/path/to/fragmentshader.vert.glsl", );
363  // add optional components of the configuration (#define, geometry shaders, ...
364  ...
365  // Register the configuration to the factory
366  Ra::Engine::ShaderConfigurationFactory::addConfiguration( myConfig );
367 ~~~
368 
369 once registered, a shader configuration could be fetched from the factory by its name :
370 ~~~{.cpp}
371 auto theConfig =
372  Ra::Engine::ShaderConfigurationFactory::getConfiguration( "ConfigName" );
373 ~~~
374 
375 ### Registering a RenderTechnique
376 A Ra::Engine::RenderTechnique describes which Ra::Engine::ShaderConfiguration a renderer will use for each of its
377 rendering passes. Such a render technique could encompass a Ra::Engine::Material but its meaning is larger than just
378 computing the BSDF.
379 In order to make a GLSL component that compute the appearance of a 3D object usable by a the default Radium renderer,
380 one must define which shader configuration to use for each pass of the renderer.
381 A Render technique will be used to configure the rendering of a geometry and the association between the geometry and
382 the render technique is made in a Ra::Engine::RenderObject.
383 Making a GLSL component available for the Ra::Engine::ForwardRenderer default renderer in Radium, on must define which
384 Ra::Engine::ShaderConfiguration to use for the passes Ra::Engine::DefaultRenderingPasses::LIGHTING_OPAQUE,
385 Ra::Engine::DefaultRenderingPasses::Z_PREPASS and, if the appearance might be transparent,
386 Ra::Engine::DefaultRenderingPasses::LIGHTING_TRANSPARENT.
387 
388 To do that, the Default render technique, the one that wil be used by Ra::Engine::ForwardRenderer must be registered
389 into the Ra::Engine::EngineRenderTechniques factory. This is done according to the following
390 
391 ~~~{.cpp}
392 Ra::Engine::EngineRenderTechniques::registerDefaultTechnique(
393  // This name will be used to query the RenderTechnique when rendering
394  "NameOfTheTechnique",
395  // This lambda will be executed to configure the rendering technique for a RenderObject
396  []( Ra::Engine::RenderTechnique& rt, bool isTransparent ) {
397  // Configure the technique to render this object at the mandatory pass
398  auto lightpass =
399  Ra::Engine::ShaderConfigurationFactory::getConfiguration( "MyTechniqueOpaque" );
400  rt.setConfiguration( lightpass, DefaultRenderingPasses::LIGHTING_OPAQUE );
401 
402  // Z prepass (Recommended) :
403  auto zprepass =
404  Ra::Engine::ShaderConfigurationFactory::getConfiguration( "MyTechniqueZprepass" );
405  rt.setConfiguration( zprepass, DefaultRenderingPasses::Z_PREPASS );
406  // Transparent pass (0ptional) : If Transparent ... add LitOIT
407  if ( isTransparent )
408  {
409  auto transparentpass =
410  Ra::Engine::ShaderConfigurationFactory::getConfiguration( "MyTechiqueTransparent" );
411  rt.setConfiguration( transparentpass,
412  DefaultRenderingPasses::LIGHTING_TRANSPARENT );
413  }
414  } );
415 ~~~
416 
417 once registered, the render technique could then be associated with any render object using the following principle :
418 ~~~{.cpp}
419 // Construct and initialize a Ra::Engine::RenderTechnique object
420 Ra::Engine::RenderTechnique rt;
421 // Associate a Ra::engine::Material with the render technique if needed
422 bool isMaterialTransparent = false;
423 if ( haveMaterial ) {
424  std::shared_ptr<Ra::Engine::Material> radiumMaterial( new MyMaterial(...) );
425  isMaterialTransparent = radiumMaterial->isTransparent();
426  rt.setMaterial( radiumMaterial );
427 } else {
428  rt.setMaterial( nullptr );
429 }
430 // configure the render technique for rendering this material with the default renderer
431 auto builder = Ra::Engine::EngineRenderTechniques::getDefaultTechnique( "NameOfTheTechnique" );
432 builder.second( rt, isMaterialTransparent );
433 ~~~

436 ## Rendering without using Materials {#non-bsdf-rendering}
437 The _Radium Material Library_ and related components are mainly designed to manage Materials as a representation of a _Bidirectional Scattering Distribution function (BSDF)_.
438 
439 When rendering, it is sometime useful to compute the final color of an object that do not rely on a bsdf but just on a specific color for each geometry fragment.
440 
441 To define a custom fragment's color computation shader and use it with application provided parameters, the following steps are required :
442 
443 1. Develop specific vertex and fragment shaders to compute the fragment color
444 2. Build a Ra::Engine::ShaderConfiguration that uses these shaders
445 3. Build a render technique that use this configuration
446 4. If the shaders have uniform parameters, implement a specific Ra::Engine::ShaderParameterProvider and associate an instance of the parameter provider to the render technique.
447 5. Associate the render technique with a geometry in a Ra::Engine::RenderObject
448 
449 Here is an example snippet.
450 ~~~{.cpp}
451 // 1. Implement a parameter provider to provide the uniforms for the shader
452 class MyParameterProvider : public Ra:Engine::ShaderParameterProvider {
453 public:
454  MyParameterProvider() {}
455  ~MyParameterProvider() {}
456  void updateGL() override {
457  // Method called before drawing each frame in Renderer::updateRenderObjectsInternal.
458  // The name of the parameter corresponds to the shader's uniform name.
459  m_renderParameters.addParameter( "aColorUniform", m_colorParameter );
460  m_renderParameters.addParameter( "aScalarUniform", m_scalarParameter );
461  }
462 
463  void setOrComputeTheParameterValues() {
464  // client side computation of the parameters, e.g.
465  m_colorParameter = Ra::Core::Color::Red();
466  m_scalarParameter = .5_ra;
467  }
468 private:
469  Ra::Core::Color m_colorParameter;
470  Scalar m_scalarParameter;
471 }
472 
473 // 2. Implement a specific vertex and fragment shaders to compute the fragment color based on uniform values
474 // Vertex shader source code
475 const std::string vertexShaderSource{
476  "#include \"TransformStructs.glsl\"\n"
477  "layout (location = 0) in vec3 in_position;\n"
478  "uniform Transform transform;\n"
479  "void main(void)\n"
480  "{\n"
481  " mat4 mvp = transform.proj * transform.view;\n"
482  " gl_Position = mvp*vec4(in_position.xyz, 1.0);\n"
483  "}\n"};
484 // Fragment shader source code
485 const std::string fragmentShaderSource{
486  "layout (location = 0) out vec4 out_color;\n"
487  "uniform vec4 aColorUniform;\n"
488  "uniform float aScalarUniform;\n"
489  "void main(void)\n"
490  "{\n"
491  " out_color = aColorUniform*aScalarUniform;\n"
492  "}\n"};
493 
494 // 3. Setup a Ra::Engine::ShaderConfiguration that uses these shaders
495 Ra::Engine::ShaderConfiguration myConfig{"MyColorComputation"};
496 config.addShaderSource( Ra::Engine::ShaderType::ShaderType_VERTEX, vertexShaderSource );
497 config.addShaderSource( Ra::Engine::ShaderType::ShaderType_FRAGMENT, fragmentShaderSource );
498 Ra::Engine::ShaderConfigurationFactory::addConfiguration( myConfig );
499 
500 // 4. Build a render technique that use this configuration
501 Ra::Engine::RenderTechnique renderTechnique;
502 renderTechnique.setConfiguration( myConfig, DefaultRenderingPasses::LIGHTING_OPAQUE );
503 
504 // 5. Create and associate the parameter provider with the RenderTechnique
505 auto parameterProvider = std::make_shared<MyParameterProvider>();
506 parameterProvider->setOrComputeTheParameterValues();
507 renderTechnique.setParametersProvider(parameterProvider);
508 
509 // 6. Associate the render technique with a geometry in a Ra::Engine::RenderObject
510 std::shared_ptr<Ra::Engine::Mesh> mesh( new Ra::Engine::Mesh( "my mesh" ) );
511 mesh->loadGeometry( Ra::Core::Geometry::makeSharpBox( {0.1f, 0.1f, 0.1f} ) );
512 auto renderObject = Ra::Engine::RenderObject::createRenderObject(
513  "myRenderObject", radiumComponent,
